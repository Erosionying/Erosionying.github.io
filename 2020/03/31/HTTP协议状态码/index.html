HTTP协议状态码

Posted on 2020-03-31

虽然在学习计算机网络的阶段我们已经了解过Http协议，不过在开发过程中很多时候难免会忘记或者误用状态码。状态码带给我们很多好处。

  • 对发生的请求错误有很好的自描述信息。
  • 根据状态码，客户端可以较清楚地知道错误出在自身还是服务端，从而进行改变请求方式或内容

1XX系列状态码

  • 信息性状态码，Web API并不使用1XX的状态码，所以这里略过…

    2XX系列状态码

    2XX系列的状态码表示客户端的请求被正确执行
  • 200 - OK，表示此次请求是成功的。
  • 201 - Created，表示请求服务端成功，服务端成功理解了客户端的请求，并创建了对应的资源。
  • 204 - No Content，请求成功，服务端也成功理解了请求，但是操作本身不该返回任何数据，例如对资源的删除操作。

    3XX系列状态码

    用于页面的跳转操作，例如告诉搜索引擎，某个页面的网址已经永久的改变了，这一系列的状态码在Web API中也几乎不会遇到，这里也直接略过…

    4XX系列状态码

    4XX系列的状态码应该是大多数人见过的最多状态码了，比如：404，表示客户端错误
  • 400 - Bad Request，表示API消费者发送到服务器的请求是有错误的
  • 401 - Unauthorized，表示没有提供授权信息或者提供的授权信息不正确。
  • 402 - Forbidden，表示身份认证已经成功，但是已认证的用户却无法访问请求的资源。
  • 404 - Not Found，表示请求的资源不存在。
  • 405 - Method not allowed，当尝试发送请求到资源的时候，使用了不被支持的HTTP方法时，就会返回405状态码。
  • 406 - Not acceptable，这表示API消费者请求的表述格式并不被Web API所支持，并且API不会提供默认的表述格式。
  • 409 - Conflict，表示请求与服务器当前状态冲突。同长指更新资源时发生的冲突，例如，当你编辑某个资源的时候，该资源在服务器上又进行了更新，所以你编辑的资源版本已经和服务器的不一致了。当然有时候也用来表示你想要创建的资源在服务器上已经存在了。它就是用来处理并发问题的状态码。
  • 415 - Unsupported media type，与406正好相反，有一些请求必须带着数据发往服务器，这些数据都属于特定的媒体类型，如果API不支持该媒体类型格式，415就会被返回。
  • 422 - Unprocessable entity，它是HTTP扩展协议的一部分。它说命服务器已经懂得了实体的Content Type，也就是说415状态码是不合适的，此外，实体的语法也是没有问题的，所以400状态码也是不合适的。但是服务器仍然无法处理这个实体数据，这时就可以返回422状态码。

    5XX系列状态码

    出现5XX系列状态码，表示服务器端出现了错误。
  • 500 - Interna Server error，表示服务器出现了内部错误。

    错误和故障

    Error and Faults – 错误和故障这点是针对与状态码的
    错误通常是由API的消费者引起的。API消费者在请求时传递的数据是不合理的，这时API就会正常的将其拒绝。
  • 对应状态码：4XX
  • API是正常工作的状态，所以并不会影响API的可用性。
    故障是指，针对一个合理的请求，API无法返回它的响应，换句话说就是API引起的问题。
  • 对应状态码：5XX
  • 会对API整体的可用性造成影响。
